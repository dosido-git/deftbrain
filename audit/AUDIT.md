# DeftBrain — Audit Convention

The point of this doc is to keep audits cheap. Anything mechanical becomes a **gate** that runs on every tool automatically. Everything left needs human eyes and is done **ride-along** — only when a tool is already open on the bench. We do not run cold, project-wide audit campaigns.

Pairs with `CONVENTIONS.md` (which defines the code patterns). This doc defines *when and what* to check.

---

## Pile 1 — Gates (automated, run on everything, always)

These never belong in a manual audit. They are zero-marginal-cost once wired into CI, so they run on all tools every commit. If a gate is green, that whole class of bug is closed without anyone reading code.

| Gate | Catches | How |
|---|---|---|
| `no-unused-vars` (error) | Dead state, dead callbacks, orphaned refs, unused imports | ESLint rule below + `--max-warnings=0` |
| `no-undef` on `backend/routes/` | Import-sweep typos, missing requires in route handlers | Add routes to lint path |
| `knip` (optional) | Orphaned files, unused exports, unused deps — things `no-unused-vars` can't see per-file | `npx knip` |
| S7.13 guard scanner | `if (!data.X)` where `X` is not a top-level key in that handler's schema | Purpose-built scanner over `backend/routes/` |
| `smoke_runtime.py` | Class A runtime failures: HTTP 500, JSON parse failures | Run route-action fixtures against live dev server |
| Babel/parse check | Syntax errors, broken JSX | `@babel/parser` parse of changed files |

### ESLint config (repo root `.eslintrc.json`)

```json
{
  "extends": "react-app",
  "rules": {
    "no-unused-vars": ["error", {
      "vars": "all",
      "args": "after-used",
      "ignoreRestSiblings": true,
      "argsIgnorePattern": "^_"
    }]
  }
}
```

### CI command

```bash
npx eslint "src/**/*.{js,jsx}" "backend/routes/**/*.js" --max-warnings=0
```

`--max-warnings=0` is the part that matters — it makes any warning fail the build. CRA's dev server only *warns* on these in the console; it never fails. The explicit `eslint` invocation in CI is what actually enforces them.

## Pile 2 — Ride-along checklist (judgment, only when the tool is open)

These produce **valid** output, so no linter fires. They require reading the code with context loaded — which is why they're done while a tool is already being worked, not as a separate task. Cheapest moment they will ever be checked.

Run this list against any tool you touch:

- **Dead features.** A feature whose logic exists but renders no UI is dead weight, not a "kept" feature — remove it. (No UI reference + nothing writes its state = delete.)
- **Themed classNames.** Every `className` uses the `c = {}` tokens. No light-only hardcoded colors. No dropped `${c.border}` (a border with no color token renders un-themed in dark mode — valid but wrong). No malformed nested templates (`` `${ `${c.card} border`}` ``).
- **No redundant class soup.** Same token repeated in one `className` (e.g. `${c.border} ... ${c.border}`) is noise — collapse it.
- **`buildFullText` dispatches on `activeTab`.** In any multi-tab tool where results from different tabs can coexist in state, copy logic must branch on `activeTab` first, and each branch must be complete (every result field present).
- **React derived flags check underlying data, not just the flag.** State updates lag one render. A value that responds to `done` must also check the underlying condition: `(timer.done || timer.remainingSec <= 0)`. Trusting a derived flag alone is the same bug class as a wrong guard key.
- **ActionButtons, not inline copy.** Copy/print/share use `CopyBtn` / `PrintBtn` / `ShareBtn` / `ActionBar` from `../components/ActionButtons`. No inline `copyToClipboard`, no `copiedField`/`copiedIndex` state, no local `CopyBtn`, no `CopyIcon` (doesn't exist).
- **Branding present.** Copy actions append `\n\n— Generated by DeftBrain · deftbrain.com`. Print templates include the branding div.
- **Emoji icons only.** No `lucide-react` imports. Config-object icons are emoji strings (`icon: '📨'`), never component references.
- **Backend route changes need a restart.** Node caches modules at startup; a route edit has no effect until the process restarts. Frontend hot-reloads; backend does not.

## When to audit

- **Gates:** every commit, every tool, automatically. Not a decision.
- **Ride-along:** every time a tool is already open for a feature change, fix, or evaluation. The pass rides on context you've already paid for.
- **Deliberate sweep:** only your 2–3 highest-traffic tools, and only as its own batch (2–4 files, per the working-batch rule). Never a 20-file campaign — that stalls feature work and re-reads files cold, defeating the purpose.

---

## Promotion rule

Any bug class that shows up **three times** becomes a candidate for promotion from Pile 2 to Pile 1 — i.e. someone writes the scanner or lint rule that catches it. (The guard-key class earned this after fake-review-detective ×2 + laundro-mat stain. The dead-code class is already covered by `no-unused-vars`.) The aim is a one-way ratchet: the manual checklist only ever gets shorter.
