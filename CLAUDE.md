# DeftBrain — Project Guide

DeftBrain (deftbrain.com) is a suite of ~128 AI micro-tools.

- **Frontend:** Create React App. Tool pages live in `src/tools/*.js`.
- **Backend:** Express / Node (CommonJS). Route handlers live in `backend/routes/*.js`. `backend/routes/index.js` auto-discovers and mounts every `.js` file in the directory (except itself) — adding a route file makes it live, no manual wiring.
- **Repo:** `dosido-git/deftbrain`.

See `audit/CONVENTIONS.md` for full patterns and worked examples. The rules below are the non-negotiables.

## Codebase rules (always follow)

1. **No `lucide-react`.** Never import from it. All icons are emoji rendered in `<span>` tags. When a config object needs an icon, use an emoji string (e.g. `icon: '📨'`), never a component reference.

2. **Shared ActionButtons.** Copy/print/share use components from `../components/ActionButtons`:
   - `CopyBtn` — labeled copy button (`content`, `label` props)
   - `PrintBtn`, `ShareBtn`, `ActionBar` — also available
   - `CopyIcon` does **not** exist; never use it.
   - Never create inline `copyToClipboard` functions, `copiedField` / `copiedIndex` state, or a local `CopyBtn`.

3. **Dark mode.** Every tool page uses `useTheme()` → `isDark` with a `c = {}` color-config object. Never hardcode light-only colors.

4. **Branding.** Copy-to-clipboard actions append `\n\n— Generated by DeftBrain · deftbrain.com`. Print templates include the branding div.

5. **Three-way sync.** A tool's frontend tab, backend route, and output renderer must stay in sync — change one, check the other two.

## Verification gates (must pass before push)

A blocking pre-push hook runs these gates in sequence. Run them after any change and iterate until all pass:

1. `npm run check:syntax`
2. `audit` — runs with `--max-warnings=0`; **zero tolerance**, one stray warning blocks the push
3. `node scripts/scan-guard-keys.js` — AST-based guard-key mismatch detector
4. `scripts/diff-audit.py` — differential name-keyed audit (`PF-*` rules eslint doesn't cover), scoped to the tool/route files the push introduces vs the remote base; blocks only on NEW issues, not pre-existing ones

Companion tooling: `scripts/fix-guard-keys.js` (codemod for guard-key fixes); `audit/audit_v2-3-2.py` (frontend warning audit); `audit/backend_audit_v1_7.py` (backend-route audit).

**Differential audit (regression check).** Some tools carry pre-existing baseline issues, so a raw audit count isn't meaningful on its own — what matters is whether *your* edit added any. Run `python3 scripts/diff-audit.py <file>...` to audit the working tree against the same file at git `HEAD` (override with `--base <ref>`) and report only NEW vs FIXED vs pre-existing issues (exit 1 if any new). It picks the right audit script by path and uses git as the pristine baseline — no manual copies to stage. This is what Gate 4 runs in the pre-push hook.

## Known gotchas (do not relearn these the hard way)

- **`audit/audit_v2-3-2.py` is name-keyed.** Deleting `handleReset`, `loadExample`, or an *active* `sessionHistory` triggers regressions. To remove such members, **array-elide** rather than delete the named symbol.
- **Python file-I/O truncation.** `io.open(path, 'w').write(fn(s))` empties the file if `fn` raises mid-transform. Always compute the result fully **before** opening the file for write.
- **Orphaned state.** Deleting a handler without also deleting the state variables it consumed creates a second wave of lint warnings. Remove the consumed state in the same pass.

## Working style

- Work in batches of **2–4 files** at a time.
- Always work on the **latest** version of a file; if unsure which that is, ask.
- When altering an existing file, **preserve all prior improvements** unless changing them is essential to the task.
- Prefer making reasonable decisions and showing working results over asking multiple clarifying questions.
